📐 Project Authoring Guide

Design principles, constraints, and editorial cadence for Wavelet projects.

What this is: the project-side companion to WORKSHEETS.md. Anyone drafting a new project should read both. Most of what makes a worksheet good also makes a project good, but the format, the time budget, and the assessment model differ in ways that change how you design tasks.

Calibrated against: the Pixel Game (design_docs/PROJECT_PIXEL_GAME.md, projects/pixel-game.json) is the running example.

Before you start

  1. Read WORKSHEETS.md. Projects re-use the same vocabulary (concept cards, validation rules, starter code, hints) and aim at the same students. The differences here are layered on top.
  2. Read every existing project JSON. Internalise the cadence of the task list and how tasks step from "fully specified" through to "make it your own".
  3. Pin down which worksheets the project comes after. A project leans on concepts the class has already covered. List the concepts you expect to use, and circle the one or two you'll need to introduce inline.
  4. Sketch the take-home artefact first. Projects exist so a student has something to show a parent. Decide what the finished .py file does, then work backwards to the tasks.

What a project is (and isn't)

A project is a multi-session deliverable the student iterates on, saves to a single .py file, and takes home. Worksheets are the floor (everyone gets through them with validated, scaffolded problems). Projects are the ceiling (something open-ended the student owns).

Project constraints

Time budget: 3 x 1-hour supervised classes

Plan tasks for a self-paced upper-primary student to complete the D-tier and C-tier work in roughly two classes, with the third reserved for the creative A-B-tier and polishing. Not a strict allocation:

  • A D-only student should still finish something they're proud of in three classes.
  • A fast student should run out of required work by mid-class 2 and spend the rest of the time on A-B creative tasks.
  • "Supervised" means a teacher is in the room to unstick a stuck student. It does not mean the teacher writes code. Scaffolding has to do most of the unblocking.

Concept budget

Default: only use concepts already taught in worksheets the class has finished. If the project genuinely needs something new, introduce it inline via a concept card placed before the first task that uses it. Keep new-concept count to one or two per project, maximum. If you want a third, you probably want a new worksheet.

Platform: laptop only

Projects are laptop-only. Worksheets target iPad too; projects don't.

  • The project page is the most code-heavy surface in the platform: multiple editors stacked vertically, plus file save/open. iPad Safari has known UX gaps (no overwrite on save, awkward keyboard for _, :, [) that punish a multi-week project.
  • Restricting to laptop unlocks real keyboard input as a design tool. Projects can use arrow keys, letter keys, Enter, etc.
  • It also means we can rely on Tab for editor indentation, copy-paste shortcuts, and dev-tools workflows during teacher demos.

Show a small "best on a laptop" notice for any student who lands on a project from an iPad, with a link to switch to a worksheet instead.

Take-home artefact

Every project produces a single .py file. The file:

  • Runs inside Wavelet. It is not a standalone Python program; it depends on the project harness. To demo at home, a parent uploads the file to the same project page on wavelet.zone and clicks Run.
  • Has a marker comment (# Wavelet ... Project) so Open can re-hydrate it, but is loose about it: a file without the marker still opens with a warning.
  • Round-trips through the editing UI. Save assembles the file; Open parses it back via Python ast.

The three-tier task model

Projects are graded by the teacher into three bands, calibrated to roughly match the bell curve of an upper-primary class. Tier badges appear on every task card.

D  Fully specified

Bottom of the class. Needs scaffolding to stay engaged.

  • Recipe-style task with a worked example in the guidance
  • Tight validation (exact: true, solution_code)
  • Concrete success criterion the teacher can eyeball
  • 5-10 minutes if the student follows the recipe
  • Strictly 1 task : 1 area

C  Specified, no recipe

Middle of the class. The bulk of students.

  • Goal stated; student picks the approach
  • Brief hints, no worked example for the main move
  • Loose validation (output_contains, behavioural rules)
  • Concept reminders inline ("use a for loop")
  • May touch multiple areas (cross-area allowed)

A-B  Unspecified (creative)

Top ~25%. Students who finish early.

  • Prompt only. "Make it your own."
  • Multiple example directions to unstick blank-page students
  • Self-check pill + function_runs_clean safety net
  • Teacher grades on what was built, not auto-checks
  • Has a dedicated freestyle area to point at

Two things to note about how this differs from worksheets:

  • Every project has tasks at every tier. A worksheet ramps difficulty over time; a project offers all three tiers throughout, so every student can engage.
  • A student "completes" a project at whatever tier they reach. D-tier students who do the D tasks and nothing else have a complete, runnable, take-home project. C and A-B work is additive on top.

Validation maps to tiers

TierTypical rulesFailure UX
D function_fills_cells with exact: true, solution_code, function_spec Green validated tick on completion, red box with diff if wrong
C function_fills_cells with exact: false, output_contains, function_runs_clean + a structural requirement; behavioural rules for cross-area tasks Green tick or "your code ran, here's what it drew"
A-B function_runs_clean paired with a self-check tick Self-checked tick on student confirmation; runtime errors surfaced on the area card

Cross-area tasks validate against whole-project behaviour, not a single function's output. Rules like after_keypress (simulate left arrow, assert state.player_x decreased) live on the task itself, because the student is free to implement the behaviour wherever they like.

Coding areas + task checklist

The central structural decision for projects. A project is not a sequence of independent problems (that's a worksheet). It is:

  1. A small number of coding areas (editors, each holding one function body) that together make up the program.
  2. A checklist of tasks that direct the student to fill in or extend those areas.

The two are decoupled: one editor can be the target of multiple tasks, and one task can touch multiple editors.

Why the split

In a worksheet, each problem is a self-contained editor with a single right answer. In a project, the student is building one program, so the editors live together and call each other. But the program is too big to give as a single empty editor (most students would freeze). Splitting into named editors with locked structure gives:

  • A safety net (locked function signatures, locked preamble).
  • An obvious shape (the student sees draw_corners, draw_border, draw_scene as the skeleton before writing any code).
  • An iteration surface (tasks point at specific editors; no navigating a 200-line file).

Task : area cardinality

The 1:1 case is the default and the right shape for almost all D-tier work. The model allows other cardinalities for higher-tier tasks:

task → area
1 task → 1 area. The default. Tier D always, most C. "Colour the corners" edits only draw_corners.
task A task B task C → area
Many tasks → 1 area. Progressive ladder inside one editor. Same draw_background editor with three tasks: D "draw a row", C "use a for loop", A-B "make it your own".
task → area 1 area 2 area 3
1 task → many areas. Cross-area. Tier C or A-B only. "Stop the player at the walls" edits all four on_*_key bodies.
task → ∅
1 task → 0 areas. Pacing checkpoint for D students. "Read the coordinate card, then tick when ready."
∅ → your_additions
0 tasks → 1 area. Pure A-B freestyle surface. The "your additions" editor at the bottom with no task pointing at it.

Rule of thumb: D-tier tasks stay strictly 1:1. Cross-area or many-to-one is reserved for C-tier and above, and the task card must list every area it touches as chips so the student knows where to type.

What goes on each card

D  Colour the corners ✓
Warm-up: draw a coloured cell in each of the four corners of the canvas. The corners are at (0, 0), (19, 0), (0, 19), and (19, 19). Use draw(x, y, 'colour').

draw_corners no errors

Area card · right column · talks about mechanics. "Called from draw_scene."

Task card (left column): tier badge, title, guidance prose, hint button, area chips for cross-area tasks, status pill. No code.

Area card (right column): locked function signature, editor body, a one-line "about this area" descriptor, an optional syntax-error badge. No prose about what to do.

Status pills: validated vs self-checked

Every task carries a status pill. There are two visually distinct kinds because teachers need to tell them apart at grading time.

✓ pass Validated. A programmatic rule earned this tick.
☑ self Self-checked. The student ticked it.
✗ fail Failing rule. Click the card for the diff.
⚠ SyntaxError Area-shaped. Body didn't compile.
- not run Not yet attempted.

Which pill kind a task uses follows the tier:

TierPill style
DValidated only. Never self-check, because D students will tick reflexively to chase completion, which defeats the validation.
CValidated by default. Self-check allowed when the success criterion is genuinely subjective (rare).
A-BUsually self-check, paired with function_runs_clean as a runtime sanity check. Validated only when a behavioural test fits.

Load-bearing details

  • One "Run Project" button. Not per-task. The project runs as one program; per-task runs are for hidden validation only.
  • Locked function signatures. Students fill in bodies. They cannot delete a def, break indentation at the function level, or nuke the structure.
  • Locked preamble. use_canvas(...), imports etc. live in a read-only chip the student can see but not edit. Editable state lives below it.
  • Concept cards inline. Same "type": "concept" block worksheets use. Place before the first task that needs the concept. Concept cards span the full width (no right-column partner).
  • Cross-area highlighting. When a Band 2 task card is focused, the area editors it touches get a subtle highlight border.

Edit-and-run rhythm iterating

The two-column model puts the canvas + Run button at the top and stacks task/area pairs below. On a real laptop screen this means students have to scroll up to Run, and they often can't see the pixel screen and the code they're editing at the same time. Worksheets don't have this problem because each problem has its own Run button right next to its editor, so the edit-test-edit loop stays local.

Recommended path

  1. Fix sticky first. Whichever layout we go with, the current "sticky stage" not actually sticking is a separate bug. Hunt down the overflow ancestor and unblock it.
  2. Ship A in the next PR as a quick quality-of-life win. Cheap, restores the worksheet rhythm.
  3. Prototype C as a follow-up. This is the real cure for the "can't see canvas and code at the same time" complaint.
  4. Land D if C lands well. Adding A on top of C is trivial once both exist.

Editorial principles

Most of WORKSHEETS.md's editorial principles apply unchanged: short content, direct tone, concrete examples, hints that unblock instead of solve, no em-dashes. Project-specific overlays:

  • Frame every task with what tier it sits in. If you can't say whether a task is D, C, or A-B, the guidance is probably ambiguous.
  • Give the student permission to skip. A-B tasks should say "if you want" or "your call" so a D-tier student doesn't feel they've failed by leaving the creative slot at pass.
  • The intro prose sells the project. Worksheet intros explain what the student will learn. Project intros explain what the student will build, and why it's cool.
  • Don't bury the deliverable. The artefact (canvas, file, screenshot) is the point; tasks are the path to it. Artefact visible first, tasks below.
  • Inline new-concept introductions are concept cards, not paragraphs. Visual distinction matters; they need to be re-readable.

Differences from worksheets (recap)

Quick reference for what a project author should not copy from WORKSHEETS.md:

  • No Run → Edit → Write → Apply cycle per concept. Projects are applications, not concept-introduction surfaces.
  • No 24-30 problem count. Aim for 6-10 tasks, plus 1-2 concept cards if introducing something new.
  • No "creative challenge as the last problem" structure. Creative (A-B) tasks are interleaved across the page, not bolted on at the end.
  • No "complete every problem" expectation. A student finishes a project at a stopping point they're happy with.
  • No iPad support. Build for laptop and keyboard.

Checklist for reviewing a new project

  • The project fits in 3 x 1-hour supervised classes for a self-paced student
  • Every task is taggable as D, C, or A-B. No ambiguous-tier tasks
  • D-tier tasks have a worked example in the guidance and tight validation
  • C-tier tasks state the goal without prescribing the approach and have loose-but-real validation
  • A-B tier tasks offer multiple example directions and use self-check pills (with function_runs_clean as a runtime safety net)
  • The required (D + C) tasks alone form a complete, runnable, take-home program
  • D-tier tasks are strictly 1 task : 1 area. Cross-area tasks are tier C or above
  • The project has a freestyle "your additions" area with at least one A-B task pointing at it
  • Status pills visibly distinguish validated ticks from self-check ticks
  • At most 1-2 new concepts are introduced inline, each via a concept card before its first use
  • Every concept used (new or recalled) is already taught in worksheets the class has finished, or covered by an inline concept card
  • Function signatures are locked in the UI; the student can't structurally break the file
  • The locked preamble holds everything the harness needs; students don't get blank-line surprises
  • The on-disk .py file demos by upload to the project page, not standalone interpretation
  • Save and Open round-trip the file faithfully (no lost code, no scrambled order)
  • The project page has a "laptop recommended" notice for iPad users
  • The intro sells the artefact in 1-3 short paragraphs
  • No em-dashes anywhere in JSON, guidance, or concept cards (use commas, colons, parentheses)
